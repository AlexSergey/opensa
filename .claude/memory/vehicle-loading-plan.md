Plan: `.claude/plans/015-vehicle-loading.md`. Parse vehicle data + place two cars statically near Tommy's spawn — **admiral** (2-colour), **camper** (4-colour). No driving yet. Deferred phases (user): **dummy** (component system / wheel spin/steer / doors / lights / seats), **collision**, **physic** (driving from handling), **vehicle_vlo** (LOD), **damage** (ok/dam).

**Files (all verified present + parse):** `static/vehicles/{admiral,camper}.dff`+`.txd` (DXT1), `static/models/generic/vehicle.txd` (19 tex rgba8888, shared); `static/data/vehicles.ide`, `carcols.dat`, `handling.cfg`.

**Formats:**
- `vehicles.ide` `cars`: `id, model, txd, type, handlingId, gameName, anims, class, frq, flags, comprules, wheelModelId, wheelScaleFront, wheelScaleRear, upgradeClass`. **admiral 0.68/0.68, camper 0.66/0.66.**
- `carcols.dat`: `col`…`end` = **palette** (`R,G,B  # idx name` per line, index by order); `car`…`end` = per-car **2-colour** combos (`name, p,s, p,s,…` palette-index pairs); `car4`…`end` = **4-colour** (`name, c1,c2,c3,c4,…`). admiral=`34,34 35,35…`; camper=`1,31,1,0…`. Names lowercased; strip `#`.
- `handling.cfg`: one line per `handlingId` = ~30 space-separated physics fields (mass, drag, dims, centre-of-mass, %submerged, traction, gears, max speed…). **Parse into a dict now (`parseHandling`); physics uses it later.**

**Vehicle DFF structure (key):** frames = body root + **dummies** (`wheel_lf/rf/lb/rb_dummy`, `chassis_dummy`, `door_*_dummy`, `bonnet/boot/bump/windscreen_dummy`, `engine`, `exhaust`, `ped_*seat`) + component atomics with **`_ok`/`_dam`** variants + **`chassis_vlo`** (LOD) + a **single `wheel` atomic** instanced at the 4 wheel dummies. **Paint markers in material colour: `(60,255,0)` = PRIMARY, `(255,0,175)` = SECONDARY** → replace with the carcol palette colour (tinting the texture). Note: generic `buildMaterial` forces white when textured → need a **vehicle-specific material builder** so paint tints show. The car DFF references textures from BOTH generic `vehicle.txd` AND its own `.txd` → **merge both texture maps**.

**Iter 3 browser fixes:** (1) **Glass transparency** — vehicle glass = material with **colour alpha 128** + texture `vehiclegeneric256` (admiral on `chassis` frame, camper on `windscreen_ok`). Shared `buildMaterial` derives transparency from `map.userData.hasAlpha` (false for vehiclegeneric256) and ignores colour alpha → glass was opaque black over the headlights. `buildVehicleMaterial` now: if `rw.color[3] < 255` → `transparent=true, opacity=alpha/255, alphaTest=0, depthWrite=false, side=DoubleSide`. (2) **Wheel size** — `WHEEL_SCALE_BOOST = 1.15` multiplier in `addWheels` (vehicles.ide scale read a bit small; can't edit the .ide). (3) **Textures were fine all along** — every DFF texture ref resolves (admiral92*/remapcamperbody256 from car txd, vehicle* + carp*/plate* from generic vehicle.txd); `map:Y` confirmed on headlights/tread/disc. The black headlights were the opaque glass panel in front. **`wheels.txd` NOT needed** — it parses to 0 textures; tread `vehicletyres128` is in generic vehicle.txd, disc `admiral92wheel64` in car txd. **Still to verify in browser:** ~~right-wheel mirror~~ (fixed), car z/heading. WHEEL_SCALE_BOOST=1.2 (accepted). **Wheel mirror:** all dummy frames are identity rotation, +X=right/−X=left; the wheel atomic is modelled facing out on the **right (+X)** side, so the **LEFT** copies get the π-about-Z spin (not the right — that put all 4 facing inward).

**Custom/modded DFF fix (BinMeshPLG material recovery):** some exporters (custom mods) write **materialIndex=0 on every face** in the triangle list and store the real per-material split only in **BinMeshPLG (0x50e)** → without it the whole model renders with material 0's texture ("все текстуры поехали"; e.g. a custom admiral showed the lights texture on 57% of faces). `parseDff` now, when a multi-material geometry's triangle list is **all-zero**, parses the geometry-extension BinMeshPLG and reassigns each face's material by keying triangles on their order-independent vertex triple (`applyBinMeshMaterials`/`forEachSplitTriangle`, handles trilist + tristrip). Stock DFFs (varied indices, e.g. admiral2) are untouched. **General parser improvement — helps the map too.** Tests: 3 cases in dff.test.ts (`buildBinMeshClump`).
